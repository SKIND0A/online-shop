# Детальный план разработки десктоп-приложения интернет-магазина "ВСЕ ДЛЯ ПК" (Go + PostgreSQL)

## 1. Цели проекта и границы MVP

### 1.1 Бизнес-цели
- Запустить десктоп-приложение магазина комплектующих и техники для ПК.
- Дать пользователю удобный каталог, корзину, оформление заказа и просмотр статуса.
- Дать администратору управление товарами, заказами, остатками и клиентами.

### 1.2 Целевая аудитория
- Частные покупатели (игровые ПК, апгрейд, периферия).
- Малый бизнес (закупка техники партиями).
- Администраторы магазина/контент-менеджеры.

### 1.3 MVP (обязательный минимум)
- Регистрация/вход пользователей.
- Каталог с категориями, фильтрами и поиском.
- Карточка товара (описание, цена, остаток, фото).
- Корзина и оформление заказа.
- История заказов.
- Админ-панель (CRUD товаров, изменение статусов заказов).
- PostgreSQL как основная БД.

### 1.4 Что отложить после MVP
- Платежные шлюзы (онлайн-эквайринг).
- Система рекомендаций.
- Программа лояльности.
- Интеграции с внешней ERP/CRM.

## 2. Выбор технического стека

### 2.1 Язык и бекенд
- Go 1.22+.
- HTTP API: `gin` или `fiber` (для старта проще `gin`).
- Конфигурация: `viper` или `cleanenv`.
- Логи: `zap` или `slog`.

### 2.2 Десктоп UI
- Вариант 1 (рекомендуется для Go): `Wails` (Go + web UI).
- Вариант 2: `Fyne` (полностью нативный Go UI, быстрее старт, проще UI).

Рекомендация:
- Если нужен современный и гибкий интерфейс -> `Wails`.
- Если нужен полностью Go-стек без фронтенд-фреймворка -> `Fyne`.

### 2.3 База данных
- PostgreSQL 15+.
- Доступ к БД: `pgx` (`jackc/pgx/v5`).
- Миграции: `golang-migrate`.

### 2.4 Инфраструктура
- Локально: Docker Compose (PostgreSQL + pgAdmin опционально).
- CI/CD: GitHub Actions (тесты, линтер, сборка).
- Линтер: `golangci-lint`.

## 3. Архитектура проекта

### 3.1 Подход
- Clean Architecture / Hexagonal (чистое разделение слоев):
  - `domain` (сущности и бизнес-правила)
  - `usecase` (сценарии)
  - `repository` (доступ к БД)
  - `delivery` (HTTP/RPC/UI handlers)

### 3.2 Модули
- Auth (пользователи, роли, сессии/JWT).
- Catalog (категории, товары, характеристики).
- Cart (корзина).
- Order (заказы, статусы, адреса).
- Admin (управление данными).

### 3.3 Структура папок (пример)
- `cmd/app` - точка входа.
- `internal/domain` - сущности.
- `internal/usecase` - бизнес-логика.
- `internal/repository/postgres` - SQL-реализация.
- `internal/delivery/http` - API.
- `internal/config` - конфиг.
- `migrations` - SQL-миграции.
- `ui` - десктоп-интерфейс (`wails` или `fyne`).

## 4. Проектирование базы данных PostgreSQL

### 4.1 Основные таблицы
- `users` (id, email, password_hash, role, created_at).
- `categories` (id, name, parent_id).
- `products` (id, category_id, sku, name, description, price, stock, is_active).
- `product_images` (id, product_id, url, sort_order).
- `carts` (id, user_id, updated_at).
- `cart_items` (id, cart_id, product_id, qty, price_snapshot).
- `orders` (id, user_id, status, total_price, created_at).
- `order_items` (id, order_id, product_id, qty, unit_price).
- `addresses` (id, user_id, city, street, house, comment).
- `audit_logs` (id, actor_id, entity, action, payload, created_at).

### 4.2 Индексы и ограничения
- Уникальные индексы: `users.email`, `products.sku`.
- Индексы на фильтруемые поля: `products(category_id, price, is_active)`, `orders(user_id, status)`.
- Внешние ключи на все связи.
- Ограничения `CHECK` (price > 0, qty > 0, stock >= 0).

### 4.3 Миграции
- Каждое изменение схемы через версионированную миграцию.
- Отдельные `up.sql` / `down.sql`.

## 5. API и контракты

### 5.1 Клиентские эндпоинты
- `POST /auth/register`, `POST /auth/login`.
- `GET /categories`, `GET /products`, `GET /products/{id}`.
- `GET /cart`, `POST /cart/items`, `PATCH /cart/items/{id}`, `DELETE /cart/items/{id}`.
- `POST /orders`, `GET /orders`, `GET /orders/{id}`.

### 5.2 Админские эндпоинты
- `POST /admin/products`, `PATCH /admin/products/{id}`, `DELETE /admin/products/{id}`.
- `GET /admin/orders`, `PATCH /admin/orders/{id}/status`.

### 5.3 Формат ответов
- Единый JSON-формат:
  - `success`
  - `data`
  - `error` (code, message, details)

## 6. Безопасность

### 6.1 Аутентификация и авторизация
- JWT Access + Refresh (или серверные сессии).
- Роли: `customer`, `admin`.
- Middleware для проверки ролей.

### 6.2 Защита данных
- Хэш паролей: `bcrypt`/`argon2`.
- SQL только параметризованный (`pgx`).
- Валидация входящих данных на каждом endpoint.

### 6.3 Практики безопасности
- Не хранить секреты в коде, только через `.env`.
- Ограничение попыток логина (rate-limit).
- Аудит действий администратора.

## 7. План реализации по этапам

### Этап 0: Инициализация (1-2 дня)
- Создать репозиторий и базовую структуру проекта.
- Настроить `go mod init`.
- Подключить линтер, форматтер, pre-commit (опционально).
- Настроить Docker Compose с PostgreSQL.

**Результат:** проект запускается, БД поднимается локально.

### Этап 1: База и миграции (2-4 дня)
- Спроектировать ER-диаграмму.
- Создать первые миграции (`users`, `categories`, `products`, `orders` и пр.).
- Реализовать слой подключения к БД и healthcheck.

**Результат:** схема БД готова и версионируется миграциями.

### Этап 2: Auth и пользователи (2-3 дня)
- Регистрация и вход.
- Хэширование паролей.
- JWT/сессии + middleware авторизации.

**Результат:** пользователь может зарегистрироваться и войти.

### Этап 3: Каталог и поиск (3-5 дней)
- CRUD категорий и товаров (минимум часть через админ API).
- Клиентские методы просмотра каталога.
- Фильтры: категория, цена, наличие.
- Поиск по названию/SKU.

**Результат:** рабочий каталог товаров.

### Этап 4: Корзина и оформление заказа (3-5 дней)
- Добавление/удаление товаров в корзине.
- Пересчет итоговой стоимости.
- Создание заказа из корзины с фиксацией цены.
- Обновление остатков на складе.

**Результат:** пользователь оформляет заказ.

### Этап 5: Админ-панель (3-4 дня)
- Интерфейсы управления товарами.
- Управление заказами и статусами (`new`, `paid`, `shipped`, `done`, `cancelled`).
- Базовая аналитика: количество заказов, выручка.

**Результат:** администратор полностью управляет магазином.

### Этап 6: Десктоп UI (5-10 дней)
- Экран авторизации.
- Главный экран каталога.
- Карточка товара.
- Корзина и оформление.
- Профиль + история заказов.
- Админ-раздел (если роль admin).

**Результат:** полноценное десктоп-приложение.

### Этап 7: Тестирование и стабилизация (3-5 дней)
- Unit-тесты usecase-слоя.
- Интеграционные тесты для репозиториев и API.
- Ручной smoke-тест всех сценариев.
- Профилирование узких мест.

**Результат:** стабильная версия перед релизом.

### Этап 8: Релиз и дистрибуция (2-3 дня)
- Сборка для Windows (`.exe`) + инсталлятор (опционально).
- Версионирование релизов.
- Подготовка пользовательской инструкции.

**Результат:** готовый дистрибутив для пользователей.

## 8. Тестовая стратегия

### 8.1 Unit-тесты
- Логика цены, корзины, валидации, статусов заказа.

### 8.2 Интеграционные тесты
- Проверка SQL-репозиториев на тестовой PostgreSQL.
- Проверка транзакций при создании заказа.

### 8.3 E2E сценарии
- Регистрация -> каталог -> корзина -> заказ -> просмотр в истории.
- Админ изменяет статус заказа, клиент видит обновление.

---

## 9. Наблюдаемость и эксплуатация

- Структурированные логи (JSON).
- Метрики (опционально): количество заказов/мин, ошибки API.
- Централизованная обработка ошибок.
- Резервное копирование PostgreSQL (ежедневно).

---

## 10. Риски и как их снизить

- Сложный UI в чистом Go -> выбрать Wails для гибкости.
- Ошибки в остатках и заказах -> транзакции + блокировки строк при списании.
- Рост базы -> индексы, пагинация, архив старых логов.
- Безопасность -> ревью auth-слоя, ограничение прав, аудит.
